Detailed Description

A document tree controller is an utility class to easily display different kind of documents(GSDocument) in an outline view.

If you need specific functionality, you can just derive from this class and overwrite the appropriate methods.

The following example creates a simple document tree controller to display scenes.

var controller = [[GSDocumentTreeController alloc] init];
[controller setDocumentTypes:["vn.scene"]];
[controller setPropertiesWindowUid:@"<properties window uid>"];
[controller setOutlineView:myOutlineView];
[controller reloadData];

As mentioned above, this is a high-level utility class, so it makes it very easy to manage and display documents in an outline-view but it has it's limits. You can overwrite a lot of methods to implement more specific behavior but if nothing helps you have to implement everything yourself from scratch. To do that, you should check out CPOutlineView class and it's protocol for data sources and delegates in Cappuccino/Cocoa documentation to implement your tree controller/data-source.

A very simple data-source looks like this:

@implementation MyTreeDataSource : CPObject
{
id _rootNode @accessors(property=rootNode);
}
// Here we have to return the child-node at "anIndex" of "anItem". If "anItem" is nil, we
// are at the top-level and access our root-node instead.
- (id)outlineView:(CPOutlineView)anOutlineView child:(CPInteger)anIndex ofItem:(id)anItem
{
return anItem ? anItem.children[anIndex] : _rootNode.children[anIndex];
}
// Here we have to return if the specified "anItem" can be expanded or not. If the item
// has any child-nodes, we return YES otherwise NO.
- (BOOL)outlineView:(CPOutlineView)anOutlineView isItemExpandable:(id)anItem
{
return anItem ? anItem.children && anItem.children.length > 0 : NO;
}
// Here we have to return the number of child-nodes of the specified items. So we just return
// the length of the "children" array of the item. If the item is nil, we know that we are at the top-level so
// we return the length of the root-node's children-array instead.
- (CPInteger)outlineView:(CPOutlineView)anOutlineView numberOfChildrenOfItem:(id)anItem
{
return anItem ? anItem.children.length : _rootNode.children.length;
}
// Here we have to return the actual value of "anItem" to display in the specified "aTableColumn".
// In this case we just return the name of the document.
- (id)outlineView:(CPOutlineView)anOutlineView objectValueForTableColumn:(CPTableColumn)aTableColumn byItem:(id)anItem
{
return [anItem.document items].name;
}
@end

Then you can set MyTreeDataSource for your outline-view like this:

var dataSource = [[MyTreeDataSource alloc] init];
// Setup your data source with data
// ......
[myOutlineView setUsesDataSource:YES];
[myOutlineView setDataSource:dataSource];

To setup a different icon depending on the displayed document type, you have to configure your CPOutlineView correctly. For each table column you can configure a different icon setup. An example looks like this:

// With setIcon:forType: we can set which icon should be displayed for what type-value. The type-value
// depends on what we have set as the type-key.
[myColumn setIcon:@"icon_dictionary" forType:@"application_language_bundle"]
[myColumn setIcon:@"icon_code_javascript" forType:@"script"]
[myColumn setIcon:@"icon_code_objj" forType:@"objj_script"]
[myColumn setIcon:@"icon_form_yellow" forType:@"DataRecordView"]
[myColumn setIcon:@"icon_form_yellow" forType:@"DataRecordCollection"]
[myColumn setIcon:@"icon_form_red" forType:@"event_command"]
[myColumn setIcon:@"icon_form_blue" forType:@"custom_view"]
// You can use "default" as type to set an icon used for all other types which have
// no specific configuration.
[myColumn setIcon:@"icon_folder_closed" forType:@"default"]
[myColumn setIcon:@"icon_folder_closed" forType:@"document_folder"]
// Set the property-name to access for the display text of the tree-node for the specified types. That
// property-name is used on the object returned by outlineView:objectValueForTableColumn:byItem: method.
[myColumn setTextKey:@"name" forTypes:["default","event_command", "application_language_bundle", "DataRecordView", "DataRecordCollection", "script", "custom_view"]];
// Set the property-name to access the type value used to calculate icon & text. We specified
// "type" here. That property-name is used on the object returned
// by outlineView:objectValueForTableColumn:byItem: method.
[myColumn setTypeKey:@"type"];
// If setIcon:forType:, setTextKey: and setTypeKey: are not enough, you can also specify a block to calculate the icon and text.
// object - The object returned in outlineView:child:ofItem: method.
// view - The GSImageAndTextView used to display icon + text for a outline-view row.
// type - The value of the attribute of "object" defined by the configured type-key.
[myColumn setFormulaBlock:function(object, view, type){
// If the tree-node has children, we want to display a folder icon, otherwise we
// display the configured icon.
if(object.children)
{
[view setImage:view._imagesByType["document_folder"]]
}
else
{
[view setImage:view._imagesByType[type]];
}
// We set the configured text but if no text-key has been defined for the type, we default to
// the "name" property. We wrap this into GSReplaceLocalizedString call to allow localization too.
[view setText:GSReplaceLocalizedString(object[view._textKeysByType[type] || "name"])];
}]

To give some examples what is possible with the document tree controller

  • The entire Extension Editor outline-view is done using this class as a base.
  • The left and right outline-view of the "Create Extension Wizard" is done using this class as a base
  • The "GSDocumentTree" control-type, used in JSON based UI Views such as "Select Scene", is done using this class.

As you can see, a lot can be done very easy just using this utility class. If it comes to "GSDocumentTree" control-type, it has a property where you can specify a custom document-tree controller class as well. Check GSDocumentTree in Extension Documentation in Help File for more information

- (void) _initContextMenu: (CPMenu)  aMenu
implementation

Initializes the context/right-click menu used for each document.

The commands "Copy", "Paste", "Delete" and "Properties" are added by default. You can overwrite this method in your derived class to add your own commands. An example looks like this:

- (void)_initContextMenu:(CPMenu)aMenu
{
var menu = aMenu;
var newMenu = [[CPMenu alloc] initWithTitle:@"New"];
var newItem = [menu addItemWithTitle:@"New" action:null keyEquivalent:@""];
[newItem setSubmenu:newMenu];
var item = [newMenu addItemWithTitle:@"New Document" action:@selector(newDocument:) keyEquivalent:""];
[item setTarget:self];
[item setIcon:@"icon_form_yellow_new"];
[item setTag:3000];
var item = [CPMenuItem separatorItem];
[menu addItem:item];
[super _initContextMenu:menu];
}

The above example will create the following context/right-click menu


  New > New Document
  -----
  Copy
  Paste
  Delete
  -----
  Properties

- (BOOL) validateMenuItem: (CPMenuItem)  aMenuItem
implementation

Validates if the specified menu item should be enabled or disabled.

The menu item with the tag 3000 is considered as the "New Document" menu item. So you don't need to overwrite this method if you only modification to the context/right-click menu is the "New Document" menu item.

If you have done more complex modification to the context/right-click menu, you have to overwrite this method with your own implementation. An example looks like this

- (BOOL)validateMenuItem:(CPMenuItem)aMenuItem
{
// Get validation result for default items
var result = [super validateMenuItem:aMenuItem];
// Get currently selected tree node
var item = [self selectedItem];
switch([aMenuItem tag])
{
case 3000:
case 3001:
case 3002:
case 3003:
case 3004:
case 3005:
case 3006:
// Enable menu item if tree node exists and is connected to a document.
result = item && item.document;
break;
}
return result;
}
