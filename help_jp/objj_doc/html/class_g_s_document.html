Detailed Description

This class together with GSDocumentManager defines the heart-piece of Visual Novel Maker's Document Management System (DMS).

Any kind of project-related data, except resources, is stored using a documents. Each document is stored as JSON in [project-dir]/data directory using its unique ID (UID) as file-name. The UID is used to uniquely identify a document even if it comes from a different project.

A document is an unsorted data-container and can store any kind of data as long as the data can be seralized to JSON. To access a document's data, use the items property. The following example show how to create a new document:

// Create a new document. The UID is automatically calculated.
var doc = [[GSDocument alloc] init];
// Fill the document with some data. You should at least provide a "name" and a "type" property.
doc.items.name = "My New Document";
doc.items.type = "com.example.my_doc_type";
doc.items.value1 = 500;
doc.items.value2 = "A Text";
doc.items.value3 = { value1: "A structure", value2: "with 2 properties" };
// Whenever you make changes to a document, you have to mark the document as unsaved
// which means that if the user saves the project, the document will be saved as well.
// You should never force to immediately save a document to disk, let the user decide when
// to save.
[doc setIsSaved:NO];

Document Summary

Each document has a so called "summary", that summary contains a very small list of fields to summarize the document or to be more clear: Those fields will always be accessible even if the document has not been loaded yet. For Example, the "name" and the "type" field are part of the summary by default, so you can always access the name and type of each document even if it hasn't been loaded yet. That is useful if you want to display lots of documents in a table-view or outline-view without loading each one.

However, don't put unnecessary fields or any big-data into the summary because that will increase memory consumption as well as start-up time. The following example shows you how to put additional fields to the summary:

// Create a new document. The UID is automatically calculated.
var doc = [[GSDocument alloc] init];
// Fill the document with some data. You should at least provide a "name" and a "type" property.
doc.items.name = "My New Document";
doc.items.type = "com.example.my_doc_type";
doc.items.someValue = "Some Value";
doc.items.orderIndex = 5;
// We add the orderIndex field to the summary so we can access it even if
// the document has not been loaded which is useful to display our documents
// in the right-order in a table-view for example.
[doc addItemToSummary:@"orderIndex"];
// Whenever you make changes to a document, you have to mark the document as unsaved
// which means that if the user saves the project, the document will be saved as well.
// You should never force to immediately save a document to disk, let the user decide when
// to save.
[doc setIsSaved:NO];

If you only make changes to fields which are part of the document's summary, you can use GSDocument::setSummarySaved: method to only mark the document's summary as unsaved. That can speed up saving time if you have a big document but only changed the order-index for example.

Temporary Documents

You can mark a document as temporary during creation to tell the DMS that this document should never be written to disk.

// Create a new temporary document.
var doc = [[GSDocument alloc] initWithUid:@"my_temp_doc" temporary:YES];
// Fill the document with some data. You should at least provide a "name" and a "type" property.
doc.items.name = "My New Document";
doc.items.type = "com.example.my_doc_type";
doc.items.someValue = "Some Value";
// Even if we mark the document as unsaved here, it will not be saved.
[doc setIsSaved:NO];

A temporary document is useful whenever you need a document object with some data but you actually don't want to save it permanently. It is not necessary to specify a unique UID since by default, the temporary document will not be added to the document index. An exception is if you use GSDocument::setIsTemporary: or if you manually add the document to the index using addDocument: (GSDocumentManager) method.

An example is the "Create Extension" wizard which you can find under Tools > Developer Tools > Create Extension. That wizward uses GSDocumentTreeController class to display all extension scripts and game scripts on the left and right side. It uses a temporary folder document to display the "Game Scripts" folder to seperate the game scripts from the editor scripts. Since "Game Scripts" actually doesn't exist in the project and is only necessary for display purposes, it is a temporary document.

Temporary Fields

There might be cases where you want to add fields to a document, like for display purposes, but actually don't want them to get saved permanently. In that case, you can mark a field as temporary using markItemAsTemporary: (GSDocument) method. A temporary field will never be saved and will be lost after the document has been reloaded. Another advantage is that a temporary field can hold non-serializable data. An example looks like this:

// Create a new document.
var doc = [[GSDocument alloc] init];
// Fill the document with some data. You should at least provide a "name" and a "type" property.
doc.items.name = "My New Document";
doc.items.type = "com.example.my_doc_type";
doc.items.someValue = "Some Value";
doc.items.someTempValue = "A temp value";
// Turn the someTempValue field into a temporary field.
[doc markItemAsTemporary:@"someTempValue"];
// Even if we mark the document as unsaved here, our temporary field will not be saved if the
// user saves the project.
[doc setIsSaved:NO];

- (id) summary
implementation

The document's summary fields.

It is an array of field/property name defined in items-object. Each field which is listed in that array will be part of the document-summary stored in document-index, the SUMMARIES.json file. You should only put fields in here if you need to access them without loading the entire document. An example could be the document's name, type, order-index to show a document in a tree- or list-view without loading it.

- (id) temporaryItems
implementation

The document's temporary items.

It is an array of field/item names defined in items-object. Each item which is listed in that array is marked as temporary. In that case, the item is not written to disk and lost if the user re-opens the project. An item which is marked as temporary can also contain non-serializable data.

You should not manually add/remove any items/fields to this array. Instead, use the markItemAsTemporary: and unmarkItemAsTemporary: methods.
